# 核心网产品 2.11 全量需求 6 月 30 日 AI Agent 交付规划

规划日期：2026-06-06

目标日期：2026-06-30

目标口径：不采用原始清单中的 SP01/SP02/未分配排期，统一按 **2026-06-30 前完成全部 34 条需求** 规划。

执行口径：AI Agent 工厂模式，不按人月、人日或传统项目组排期估算。所有需求拆为可并行 ADU（Agentic Delivery Unit），由 Agent 自动领取上下文、生成实现、测试、复审和证据包。

## 一、硬前提

要在 2026-06-30 前完成全部 34 条需求，必须接受以下前提：

| 前提 | 要求 |
| --- | --- |
| 并发规模 | 同时运行 40-70 个专用 Agent，分布在代码、测试、文档、联调、证据、发布 6 类 lane。 |
| 执行时间 | 24 小时滚动执行，不按工作日/工作时间排期。 |
| Scope 冻结 | 6 月 6 日当天冻结需求解释，不再允许新增需求或扩大验收口径。 |
| 外部资源 | 6 月 8 日前必须给出 4G/5G UE/RAN/IMS、达梦、加密卡、国密 USIM、基站 SM4 联调资源；否则相关项只能按 mock/simulator 完成。 |
| 验收方式 | 每条需求必须有 evidence bundle：测试日志、接口样例、配置样例、抓包/运行态输出、限制说明。 |
| 冲突管理 | 代码主线按 lane 隔离，所有跨 lane 接口先生成 contract，再允许实现。 |

如果上述前提不满足，6 月 30 日仍可形成“全量需求代码包 + 模拟验收 + 未联调限制说明”，但不能宣称所有硬件/现场依赖型需求完成真实商用验收。

## 二、全量需求压缩策略

### 2.1 不再按版本拆分

34 条需求统一拆为 6 个交付群：

| 群组 | 覆盖需求 | 目标 |
| --- | --- | --- |
| A. 可观测与 OAM | KPI、4G 状态/告警/KPI、OAM、IMS 告警、LMT、链路检测 | 统一状态、指标、告警、北向接口和最小 UI。 |
| B. 5G-LAN 与 HA | 5G-LAN 配置简化、实现优化、双活 HA、工业增强网关接口、工业协议代理接口 | 把当前 5G-LAN/HA 分支能力产品化，并给新产品接口留钩子。 |
| C. 策略与 License | License、SIM 子网、QoS 动态生效、4G/5G 互操作 | 完成策略控制、授权、地址池、QoS 和互操作最小闭环。 |
| D. 安全国产化 | 加密卡、达梦 HA、ARPF、EIR、国密 UDM/USIM、SBI TLS、信令分析、违规外联、N3 SM4 | 完成 provider/interface/服务实现，硬件项同步联调。 |
| E. 架构可靠性 | 多线程、zk 异步、单机备份恢复、zk 内存池/OMU 命令 | 以可控试点和可回滚方式完成架构增强。 |
| F. 新产品与分析 | 用户业务数据分析服务器、独立工业增强网关、工业协议代理 | 以独立服务/边缘组件方式交付，不塞入核心网主流程。 |

### 2.2 完成定义

每条需求必须达到至少一种完成等级：

| 等级 | 定义 | 可用于 6/30 宣称 |
| --- | --- | --- |
| L3 真实验收 | 代码、部署、真实环境联调、验收证据齐全。 | 可以宣称完成。 |
| L2 模拟验收 | 代码完成，使用 mock/simulator/容器环境验证，外部资源未到位。 | 可以宣称软件侧完成，不能宣称现场验收完成。 |
| L1 方案 + PoC | 技术方案、接口、PoC 完成，但未形成发布代码闭环。 | 不能算完成，只能算预研完成。 |

本规划目标是：所有非外部依赖项达到 L3；硬件/现场依赖项至少达到 L2，并争取 L3。

## 六、关键风险与强制处理

| 风险 | 影响 | 强制处理 |
| --- | --- | --- |
| 外部硬件不到位 | 加密卡、国密 USIM、SM4 N3、达梦 HA 无法 L3 | 6/8 前没有资源则自动切 L2 mock，并生成限制说明。 |
| 4G/5G/IMS 环境不到位 | 互操作、IMS 告警无法 L3 | 使用仿真/日志注入完成 L2，真实环境作为补证 lane。 |
| 多线程引入稳定性风险 | 可能拖垮主线 | 只允许 feature flag + 单路径试点，不影响默认路径。 |
| OAM 范围膨胀 | 拖慢全量交付 | OAM 只做状态/指标/告警/北向，不做完整平台化。 |
| 新产品需求过大 | 工业网关、分析服务器、协议代理无法商用品质 | 以独立服务 MVP + 接口 + PoC 验收，不侵入核心网主线。 |
| Agent 冲突过多 | 集成失败 | contract-lane 先冻结接口；同一冲突域只允许一个实现 ADU 合并。 |

## 七、发布门禁

6 月 30 日全量交付必须输出：

1. 34 条需求验收矩阵，逐条标注 L3/L2/L1。
2. 每条需求 evidence bundle。
3. 全量版本包和部署脚本。
4. API 文档、配置文档、测试报告、已知问题。
5. 外部依赖未达到 L3 的限制说明。
6. 所有 feature flag 默认策略。
7. 回滚方案。

发布判定：

| 判定 | 条件 |
| --- | --- |
| 全量真实完成 | 34 条需求全部 L3。 |
| 软件侧全量完成 | 无硬件/现场资源项达到 L2，其余全部 L3。 |
| 风险发布 | 任何核心主线需求低于 L3，或外部依赖项低于 L2。 |
| 不建议发布 | KPI、4G 基础能力、5G-LAN、OAM、安全基础任一主线失败。 |

## 八、最终规划建议

如果必须在 2026-06-30 前完成全部 2.11 需求，不能按传统项目计划慢慢排期，必须采用下面的压缩打法：

1. **24 小时内冻结全部接口和验收断言。**
2. **72 小时内六大群组全部跑通端到端纵切。**
3. **7 天内主体代码全部进入 reviewed。**
4. **12 天内全部需求进入 evidenced 或外部依赖 L2。**
5. **第 13-20 天只做集成、回归、长稳和硬件补证。**
6. **第 21-24 天只做 RC、验收回放、文档和发布锁定。**

这是一套高压 AI Agent 工厂计划。它能把软件实现压缩到数天级，但外部硬件、真实 RAN/UE/IMS、数据库 HA 和国密卡联调仍是物理世界约束，必须在 6 月 8 日前同步到位，否则只能按 L2 软件侧完成。
