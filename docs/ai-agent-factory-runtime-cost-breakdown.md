# AI Agent 工厂月度运行成本明细

评估日期：2026-06-07

目标：解释通用 5G 核心网 AI Agent 工厂中，`worker / 存储 / CI` 和 `Agent 平台维护` 两类月度运行成本具体花在什么地方。

## 一、先给结论

如果已有本地服务器、现成 Codex/Agent 工具、已有 CI 和制品仓库，那么这两项现金成本可以明显降低。

之前估算的生产可用版：

| 成本项 | 月度估算 |
| --- | ---: |
| 4-8 台 worker / 存储 / CI | 2万-8万 RMB/月 |
| Agent 平台维护 | 1万-4万 RMB/月 |

更准确地说：

| 运行方式 | worker / 存储 / CI | Agent 平台维护 | 说明 |
| --- | ---: | ---: | --- |
| 本地已有机器 + 手工编排 | 0.3万-1.5万/月 | 0.3万-1万/月 | 适合 MVP，不适合高并发发布。 |
| 本地服务器 + 自建轻量平台 | 0.8万-3万/月 | 0.8万-2万/月 | 推荐早期路线。 |
| 云主机 + 生产 CI + 自动证据 | 2万-8万/月 | 1万-4万/月 | 之前文档采用的口径。 |
| 企业多团队 + 实验室 | 8万-30万+/月 | 5万-20万+/月 | 多版本、多项目、多环境。 |

所以这两项不是固定成本，而是取决于是否云上运行、是否 24 小时并发、是否有长稳/压测/pcap/core dump、是否需要企业级审计。

## 二、worker / 存储 / CI 花在哪里

worker 不是给 Agent “聊天”用的，而是给 Agent 生成的代码做真实工程验证。

### 2.1 worker 类型

| worker 类型 | 数量 | 主要用途 | 月度成本去向 |
| --- | ---: | --- | --- |
| build-worker | 1-2 | Open5GS C/C++ 编译、Meson/Ninja、静态检查、ASan/UBSan。 | CPU、内存、磁盘 IO、编译缓存。 |
| unit-worker | 1-2 | CUnit、pytest、Node/TypeScript 单测、mock 服务。 | 并发测试、依赖缓存、短任务调度。 |
| integration-worker | 2-3 | Open5GS + MongoDB + UERANSIM/gNBSim + NMS 集成。 | 多容器、网络 namespace、TUN/TAP、日志。 |
| chaos-worker | 1 | HA、进程重启、网络故障、DB failover、长稳。 | 长时间占用、监控、pcap、core dump。 |
| release-worker | 0-1 | 打包、制品签名、SBOM、发布回放。 | 制品生成、版本归档。 |

### 2.2 具体成本项

| 成本项 | 说明 | 生产版月度估算 |
| --- | --- | ---: |
| 计算资源 | 4-8 台 8-16 vCPU、16-64GB 内存机器，按需或包月。 | 1万-5万 |
| 长稳/压测占用 | HA、UPF、QoS、数据面需求需要长时间跑，机器不能随用随关。 | 0.5万-2万 |
| 存储 | build artifacts、日志、pcap、core dump、覆盖率、证据包。 | 0.2万-1万 |
| 网络和带宽 | 镜像拉取、制品上传、pcap 下载、跨 worker 调度。 | 0.1万-0.5万 |
| CI runner 管理 | runner 注册、队列、缓存、失败重跑、环境清理。 | 0.2万-1万 |
| 镜像和依赖缓存 | Open5GS 依赖、Node modules、Docker images、编译缓存。 | 0.1万-0.5万 |
| 监控日志 | worker 健康、磁盘、CPU、失败率、任务队列。 | 0.1万-0.5万 |
| 合计 | 云上生产口径。 | 2万-8万 |

### 2.3 为什么 5GC worker 比普通 Web 项目贵

| 原因 | 影响 |
| --- | --- |
| Open5GS 需要 Linux 网络能力 | TUN/TAP、network namespace、路由、iptables、抓包。 |
| 集成测试不是纯单测 | AMF/SMF/UPF/MongoDB/RAN/UE 仿真要一起跑。 |
| pcap/core dump 很大 | 证据和调试文件会快速占用存储。 |
| 长稳和 Chaos 会占机器 | HA、DB failover、UPF 压测不能几分钟结束。 |
| Agent 会高频触发 CI | 每个 ADU 都可能多轮 build/test/debug。 |

### 2.4 可以怎么降

| 降本方式 | 影响 |
| --- | --- |
| 本地已有服务器优先 | 现金成本可降 50%-80%。 |
| worker 自动关停 | 非长稳任务用完即停。 |
| 分层测试 | PR 只跑局部测试，夜间跑全量回归。 |
| pcap/core 生命周期 | 只保留关键失败证据。 |
| Docker 镜像和编译缓存 | 减少重复下载和重复编译。 |
| L2 mock 优先 | 减少真实环境占用。 |

## 三、Agent 平台维护花在哪里

Agent 平台维护不是买一个“AI 平台会员”的费用，而是保证 Agent 工厂不失控的持续工程工作。

### 3.1 维护对象

| 模块 | 维护内容 |
| --- | --- |
| ADU registry | 需求拆分、状态流转、依赖关系、优先级、owner、L2/L3 状态。 |
| Context Pack | 代码地图、接口说明、测试入口、风险清单随代码变化更新。 |
| Contract | API/schema/config/metrics/alarm/license/DB/crypto 合约版本管理。 |
| Prompt 模板 | 不同 Agent 的提示词、失败策略、输出格式、可写边界。 |
| Agent 队列 | 并发槽位、冲突域、任务分配、失败重试、人类升级。 |
| Review Agent | contract review、risk review、evidence review 规则和误报调整。 |
| Evidence harness | 日志收集、命令记录、pcap/core/截图/配置归档、验收矩阵。 |
| 成本控制 | token budget、模型路由、缓存命中率、异常消耗报警。 |
| 权限和安全 | secret 扫描、脱敏、目录白名单、审计日志。 |
| 工具升级 | 模型、CLI、依赖、CI runner、Docker image、测试工具升级后的回归。 |

### 3.2 具体成本项

| 成本项 | 说明 | 生产版月度估算 |
| --- | --- | ---: |
| 平台 owner 时间 | 维护队列、处理失败任务、调模型路由、处理 Agent 异常。 | 0.5万-1.5万 |
| Context/Contract 更新 | 代码变化后更新上下文包、接口契约、测试入口。 | 0.2万-1万 |
| Prompt 和策略迭代 | 修正 Agent 跑偏、误改、漏证据、重复读仓等问题。 | 0.2万-0.8万 |
| Review/Evidence 规则维护 | 降低误报漏报，让人类只看高风险项。 | 0.2万-0.8万 |
| 安全和审计 | secret 扫描、权限白名单、日志脱敏、合规记录。 | 0.2万-0.8万 |
| 平台工具/服务费用 | 看板、队列、日志、对象存储、CI 插件、商业 IDE 席位。 | 0.2万-1万 |
| 合计 | 自建轻量生产口径。 | 1万-4万 |

### 3.3 什么时候可以接近 0

如果满足以下条件，Agent 平台维护现金成本可以很低：

| 条件 | 说明 |
| --- | --- |
| 只有 1-2 个 Agent 并发 | 人工派任务即可。 |
| 需求少且低风险 | 不需要复杂 ADU 队列。 |
| 不追求自动证据 | 人工整理日志和验收。 |
| 不做多版本并行 | 不需要复杂冲突域和发布矩阵。 |
| 不做企业审计 | 不需要完整权限和审计链。 |

但这时它更像“AI 辅助开发流程”，还不是“AI Agent 工厂”。

## 四、这两项和 token 的关系

token 是 Agent 的“脑力”成本；worker 和平台是把脑力变成可发布代码的“生产线”成本。

| 没有 worker / 平台 | 后果 |
| --- | --- |
| 有 token 但没有 worker | Agent 能写代码，但不能持续编译、跑集成、抓 pcap、出证据。 |
| 有 token 但没有平台 | Agent 可以单点完成任务，但多需求并发会混乱。 |
| 有 token 但没有 evidence | 代码可能可用，但很难发布和验收。 |
| 有 token 但没有 Review Agent | 人类会被大量低质量 PR 淹没。 |

如果目标只是让 AI 写代码，token 是大头。  
如果目标是发布 5GC 功能，worker、CI、证据、测试环境和平台治理一定会出现。

## 五、推荐的精简版运行成本

如果目标是先用最少成本跑起来，建议按这个配置：

| 项目 | 配置 | 月度现金成本 |
| --- | --- | ---: |
| worker | 1 台 build + 1 台 integration，可本地机器或低成本云。 | 0.3万-1.2万 |
| 存储 | 500GB-1TB，日志生命周期 7-14 天。 | 0.05万-0.3万 |
| CI | 自建 runner + 夜间全量回归。 | 0.1万-0.5万 |
| Agent 平台 | 文件型 registry + 脚本 + 手工看板。 | 0.2万-0.8万 |
| 监控 | 基础日志和预算报警。 | 0.05万-0.2万 |
| 合计 | MVP/早期生产。 | 0.7万-3万/月 |

这比之前的 12万-35万/月低很多，因为它不包含 3-4 人 HITL，也不包含外部补证和企业实验室。

如果只看 `worker + Agent 平台维护` 两项，早期可以压到：

> **0.7万-3万 RMB/月**

生产常态再扩到：

> **3万-12万 RMB/月**

只有企业多团队、多版本、多实验室时，才需要：

> **10万-50万+ RMB/月**

## 六、建议修正成本口径

后续建议把运行成本拆成三类，不再混在一起：

| 类别 | 是否必须 | 说明 |
| --- | --- | --- |
| AI 消耗成本 | 必须 | token、工具调用、容器执行。 |
| 工程执行成本 | 必须 | worker、CI、存储、仿真、证据。 |
| 人类和实验室成本 | 视目标而定 | HITL、供应商、硬件、真实 L3 环境。 |

如果只讨论 AI Agent 工厂基础运行，不含人类和实验室，生产前期合理预算应是：

> **token 0.2万-1.5万/月 + worker/platform 0.7万-3万/月**  
> 合计约 **0.9万-4.5万/月**。

如果要稳定发布 5GC 功能，再叠加 HITL 和实验室资源。

## 七、进一步修正：哪些其实应该算 token

把明细拆开后可以看到，`Agent 平台维护` 中相当一部分并不是传统意义上的人工运维，而是 Agent 可以持续执行的工作。因此更准确的成本口径应该是：

| 原归类 | 更准确归类 | 原因 |
| --- | --- | --- |
| ADU registry 维护 | token / Agent 执行 | 需求拆分、状态整理、依赖分析可以由 Orchestrator Agent 完成。 |
| Context Pack 更新 | token / Agent 执行 | 代码地图、测试入口、风险摘要主要是上下文理解和摘要生成。 |
| Contract 更新 | token + 少量 HITL | schema/API/config 可由 Agent 草拟，人类只做关键签核。 |
| Prompt 策略迭代 | token + 少量 HITL | Agent 可基于失败样本自改模板，但高风险策略需人类确认。 |
| Review Agent | token | contract/risk/evidence review 本身就是模型调用。 |
| Evidence harness 内容整理 | token + 存储 | 日志摘要、验收矩阵生成是 token；原始日志/pcap 保存是存储。 |
| 成本控制分析 | token + 少量平台服务 | 异常 token 消耗分析可由 Agent 做，预算阈值是平台配置。 |
| 安全扫描结果解释 | token + 工具执行 | 扫描工具本身是计算，解释和修复建议是 token。 |

真正不应该算 token 的部分主要是：

| 非 token 成本 | 说明 |
| --- | --- |
| 编译和测试执行机器 | C/C++ 编译、容器启动、UERANSIM/gNBSim、长稳、压测。 |
| 存储 | 日志、pcap、core dump、制品、证据包、镜像缓存。 |
| CI runner 基础设施 | runner 进程、队列服务、缓存、调度器、网络流量。 |
| 真实实验室 | RAN/UE/IMS/达梦/加密卡/国密 USIM/SM4 对端。 |
| 人类门禁 | 架构、安全、发布承诺、L2/L3 判定。 |

因此，后续估算应把 `Agent 平台维护` 拆掉，不再作为一个大额独立项，而拆成：

| 新口径 | 月度成本 |
| --- | ---: |
| Agent/token 执行成本 | 取决于需求吞吐，通常是主要可变成本。 |
| 工具/worker 执行成本 | 0.5万-5万/月，随测试强度变化。 |
| 存储/证据成本 | 0.1万-1万/月，随 pcap/core/长稳日志变化。 |
| 少量平台服务 | 0.1万-1万/月，看板、队列、权限、对象存储、告警。 |
| HITL | 视发布承诺而定。 |

更符合 AI Agent 工厂逻辑的精简成本公式：

```text
月度成本 =
  token / Agent 执行成本
  + worker / CI / 工具执行成本
  + 存储 / 证据成本
  + 少量平台服务成本
  + 必要 HITL
  + 真实实验室资源
```

换句话说，如果 Agent 工厂做得足够彻底，很多“平台维护”确实会继续变成 token 消耗，而不是固定人工运维成本。

## 八、为什么 worker、存储、CI 不算 token

token 是模型读写文本、多模态内容和进行推理时的计费单位。worker、存储、CI 基础设施虽然由 Agent 触发，但它们消耗的是 CPU、内存、磁盘、网络、容器运行时间和测试设备时间，因此不应直接归入 token。

### 8.1 worker 算力

Agent 可以决定“运行编译”或“跑集成测试”，但真正执行的是 worker。

| 动作 | token 部分 | 非 token 部分 |
| --- | --- | --- |
| 分析编译失败 | 模型读取 build log，推理根因，生成 patch。 | 无。 |
| 执行 `ninja -C build` | Agent 发起命令的 prompt 很少。 | CPU、内存、磁盘 IO、编译缓存、进程时间。 |
| 跑 UERANSIM/gNBSim | Agent 生成场景和解释日志。 | 多进程、网络 namespace、TUN/TAP、容器、虚拟网卡。 |
| 跑 UPF 压测 | Agent 设计压测和分析结果。 | 流量生成、CPU、网卡、内核网络栈、长时间机器占用。 |
| 跑 Valgrind/ASan | Agent 解读报告。 | 测试进程慢速执行、内存和 CPU 消耗。 |

类比：Agent 像调度员和工程师；worker 是机床和试验台。调度员说的话按 token 计费，机床转起来消耗的是电、机器和时间。

### 8.2 存储

Agent 可以生成日志摘要，但原始证据必须保存。

| 数据 | 是否 token | 说明 |
| --- | --- | --- |
| build log 摘要 | 是 | 模型读取和压缩日志。 |
| 原始 build log 文件 | 否 | 存在磁盘/对象存储。 |
| pcap 摘要 | 是 | 模型或工具分析抓包结果。 |
| 原始 pcap | 否 | 可能很大，需要长期保存用于验收和复盘。 |
| core dump 解释 | 是 | 模型读 backtrace、寄存器、栈信息。 |
| core dump 文件 | 否 | 二进制文件，存储成本和下载成本。 |
| release bundle | 否 | 制品、镜像、配置、证据包归档。 |

如果所有证据都只转成文本摘要给模型，不保存原始文件，token 成本会上升，但发布可信度会下降。5GC 功能验收通常需要原始日志、pcap、配置和版本包可回放。

### 8.3 CI 基础设施

CI 是把 Agent 产物变成可验证工程产物的执行系统。

| CI 能力 | 非 token 消耗 |
| --- | --- |
| runner 队列 | 调度服务、runner 进程、锁、并发槽位。 |
| 环境准备 | 拉镜像、装依赖、启动数据库和仿真器。 |
| 失败重跑 | 重复执行编译、测试、清理、收集日志。 |
| 缓存 | Docker layer、npm cache、ccache、Meson build cache。 |
| 制品归档 | build artifact、coverage、SBOM、release package。 |
| 网络 | 镜像、依赖、制品和日志上传下载。 |

Agent 可以维护 CI 配置，这部分是 token；但 CI 每一次实际运行都不是 token。

### 8.4 如果使用模型厂商的容器工具

如果 worker 不是自建 Linux 机器，而是使用模型厂商提供的容器/代码执行工具，那么账单上可能表现为“工具调用费”或“容器会话费”。这仍然不是 token，因为它按容器资源或会话计费，而不是按输入输出 token 计费。

因此更准确的分类是：

| 成本 | 计费对象 |
| --- | --- |
| token | 模型输入、缓存输入、输出。 |
| tool/container | 代码执行环境、容器会话、工具调用。 |
| worker | 自建或云主机 CPU、内存、磁盘、网络。 |
| storage | 日志、pcap、core、制品、镜像、证据。 |
| CI | runner、队列、缓存、制品归档、环境清理。 |

### 8.5 什么时候它们会“看起来像 token”

如果使用全托管 Agent 平台，对方可能把模型、容器、工具、存储、CI 都打包成一个统一价格。这时账单上可能看起来像“AI 使用费”，但底层仍然包括两类资源：

1. 模型推理资源，也就是 token。
2. 工程执行资源，也就是 worker、容器、存储、CI、网络。

对 5GC 来说，后者不能消失，因为代码必须被编译、仿真、压测、抓包、归档和回放。
