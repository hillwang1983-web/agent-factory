# AI Agent 工厂最小 HITL 人员评估

评估日期：2026-06-07

关联文档：

- `docs/ai-agent-development-environment-from-zero.md`
- `docs/ai-agent-factory-cost-estimate.md`
- `docs/core-network-2.11-8human-hh-estimate.md`

评估问题：能否减少人类 HITL？能否不用 8 个人？最少几个人可以维持 AI Agent 工厂正常运行？

## 一、结论

可以不用 8 个人。

但要区分三种口径：

| 口径 | 最少人数 | 是否推荐 | 适用场景 |
| --- | ---: | --- | --- |
| 小规模试点 | 2 人 | 可行 | 2-5 个需求、L2 mock、低并发 Agent。 |
| 工厂正常运行 | 3 人 | 勉强可行 | 10-20 个 ADU 并发、以 L2 软件侧为主。 |
| 2.11 全量冲刺 | 4 人 | 推荐下限 | 34 条需求、40-70 Agent 槽位、6/30 交付。 |
| 更稳妥生产运行 | 5-6 人 | 推荐 | 包含安全国产化、HA、达梦、外部硬件补证。 |
| 原 8 人模式 | 8 人 | 高冗余/低单点风险 | 专家分工清晰，适合全量 L3 和客户验收压力大时。 |

最终建议：

> **维持 AI Agent 工厂正常运行：最低 3 人。**  
> **支撑 2.11 全量 6/30 交付：最低 4 人。**  
> **如果要尽量 L3 真实验收：建议 5-6 人。**

1 人不建议。因为 AI Agent 工厂至少需要“执行编排”和“独立验收”两个互相制衡的角色，否则很容易出现 Agent 产物看似完成、实际没人能独立确认的情况。

## 二、为什么 8 人可以压缩

原 8 人模型是按领域专家拆分：

| 原角色 | 职责 |
| --- | --- |
| H1 | 总架构 / Orchestrator Reviewer |
| H2 | 5GC 控制面 |
| H3 | UPF / 5G-LAN / HA |
| H4 | EPC / 互操作 / IMS |
| H5 | OAM / NMS / 数据库 |
| H6 | 安全国产化 |
| H7 | QA / 环境 / Chaos |
| H8 | 发布 / 验收 / 产品裁决 |

在 AI Agent 工厂中，以下工作可以被 Agent 或自动化吸收：

| 原人类工作 | 可替代方式 |
| --- | --- |
| 普通代码实现 | Developer Agent + TestWriter Agent。 |
| 编译错误修复 | BuildAgent。 |
| 常规单测/接口测试失败定位 | TestRunner + Debugger Agent。 |
| 普通文档和证据整理 | Evidence Agent。 |
| 常规代码 review 第一轮 | Review Agent + 静态检查 + contract check。 |
| 测试报告汇总 | evidence harness。 |

因此人类只保留不可替代职责：

| 不可替代职责 | 原因 |
| --- | --- |
| 架构和 contract 最终裁决 | 错误 contract 会让所有 Agent 一起跑偏。 |
| 核心网协议/数据面风险判断 | 5GC/UPF/HA/互操作错误可能不是单测能发现的。 |
| 安全/国产化/数据库/外部资源边界 | 涉及密钥、License、供应商、真实环境。 |
| 发布口径和 L2/L3 判定 | 这是产品和客户承诺，不能交给 Agent 自判。 |
| P0/P1 根因最终确认 | Agent 可以定位，但重大风险需要人类签核。 |

## 三、4 人最小推荐配置

如果目标仍然是 2.11 全量 34 条需求在 2026-06-30 前交付，推荐最小团队为 4 人。

| 角色 | 合并原 8 人职责 | 主要负责 | 不能缺的原因 |
| --- | --- | --- | --- |
| P1 Agent 工厂负责人 / 总架构 | H1 + H8 | ADU 队列、contract freeze、scope 裁决、发布门禁。 | 没有 P1，Agent 并发会失控，需求口径会漂移。 |
| P2 核心网协议与数据面专家 | H2 + H3 + H4 部分 | AMF/SMF/UPF/PFCP/5G-LAN/HA/QoS/互操作。 | 这是 2.11 技术风险最高的主线。 |
| P3 平台、安全与数据专家 | H5 + H6 | OAM/NMS、达梦、License、ARPF/EIR、TLS、国密、加密卡。 | 安全和外部依赖不能完全由 Agent 自判。 |
| P4 QA / 环境 / 发布证据专家 | H7 + H8 部分 | CI、仿真、Chaos、L2/L3 证据、验收矩阵、回归门禁。 | 没有独立质量门禁，Agent 产物容易“看起来完成”。 |

这 4 个人不是 4 个全职程序员，而是 4 个高密度 HITL owner。普通编码、普通 debug、普通测试和文档整理仍由 Agent 完成。

## 四、3 人极限配置

3 人可以维持工厂正常运行，但适合 L2 软件侧为主，不适合全量 L3。

| 角色 | 合并职责 | 风险 |
| --- | --- | --- |
| P1 Agent 工厂 / 架构 / 发布 | H1 + H8 | P1 会成为所有 contract 和发布裁决单点。 |
| P2 核心网协议 / UPF / HA | H2 + H3 + H4 | 控制面、数据面、互操作都压到一个人，审查负载高。 |
| P3 平台安全 / QA / 环境 | H5 + H6 + H7 | 安全、DB、OAM、测试环境合并，容易漏审。 |

3 人模式的前提：

| 前提 | 要求 |
| --- | --- |
| 交付等级 | 以 L2 软件侧全量为主，关键主线少量 L3。 |
| 外部资源 | 达梦、加密卡、国密、SM4、IMS 等不到位时直接 L2。 |
| Agent 规则 | 失败超过 2-3 轮必须升级，不能无限自愈。 |
| Contract | 冻结后严禁随意变更。 |
| Scope | OAM、新产品、DPI、全量双活不能扩展。 |

结论：3 人可以“维持工厂”，但不建议用于高承诺的 2.11 全量 L3 交付。

## 五、2 人试点配置

2 人只能用于试点或低风险维护。

| 角色 | 职责 |
| --- | --- |
| P1 Agent 工厂 / 架构 / 发布 | 调度 Agent、冻结需求、review 关键 PR、发布结论。 |
| P2 技术 / QA / 环境 | 核心网技术判断、测试环境、证据确认。 |

适用：

| 可做 | 不建议做 |
| --- | --- |
| 2-5 个需求试点。 | 34 条需求全量冲刺。 |
| L2 mock 验收。 | 多个外部硬件 L3 联调。 |
| 单 lane 或双 lane 并发。 | 40-70 Agent 槽位全速并发。 |
| 建设工厂 MVP。 | 发布客户承诺版本。 |

2 人模式最大问题是没有足够的独立复核。P1 和 P2 任一人判断错误，Agent 工厂会把错误快速放大。

## 六、1 人为什么不建议

1 人理论上可以“启动”Agent 工厂，但不应该视为正常运行。

| 问题 | 后果 |
| --- | --- |
| 没有独立 review | Agent 产物很容易被同一个人误判为完成。 |
| 架构和验收冲突 | 同一个人既定规则又放行结果，门禁失效。 |
| P0/P1 无人交叉确认 | 复杂 crash、并发、数据面、安全问题风险高。 |
| 外部资源协调被挤占 | 人会被环境问题拖走，Agent 队列无人维护。 |
| 发布口径不可信 | L2/L3 判定缺少第二视角。 |

1 人只适合做本地验证、工具试跑、单需求 PoC。

## 七、人数减少后的 HH 变化

减少人数不等于总 HH 按比例减少。真正能减少 HH 的是：

| 手段 | HH 影响 |
| --- | --- |
| Review Agent 先筛一轮 | 减少 20%-30% 人类 review。 |
| evidence bundle 标准化 | 减少 15%-25% 验收整理。 |
| contract freeze 严格执行 | 减少 20%-35% 返工裁决。 |
| L2 mock 优先 | 减少 20%-40% 外部联调等待和排障。 |
| 失败 3 轮升级 | 减少无限 debug 消耗。 |

### 7.1 推荐 4 人模式 HH

| 交付口径 | 原 8 人 HH | 4 人优化后 HH | 平均每人 |
| --- | ---: | ---: | ---: |
| L2 软件侧全量 | 280-380 | 180-260 | 45-65 |
| L2 全量 + 关键 L3 | 360-520 | 240-360 | 60-90 |
| 尽量全量 L3 | 520-760 | 380-560 | 95-140 |

4 人模式能降低总 HH，但每个人强度更高。它依赖自动化门禁足够强，否则总 HH 会反弹。

### 7.2 3 人模式 HH

| 交付口径 | 3 人 HH | 平均每人 | 备注 |
| --- | ---: | ---: | --- |
| L2 软件侧全量 | 170-250 | 57-83 | 可行。 |
| L2 全量 + 少量 L3 | 230-340 | 77-113 | 勉强可行。 |
| 尽量全量 L3 | 360-520 | 120-173 | 不推荐，单点过重。 |

3 人不是更省，而是更冒险。因为很多领域审查会合并到同一个人身上，决策速度可能提高，但漏审概率也提高。

## 八、减少 HITL 的机制

要真正减少人类 HITL，需要把“人类检查”前置成“机器门禁”。

### 8.1 自动门禁

| 门禁 | 自动化方式 |
| --- | --- |
| Contract check | schema、OpenAPI、配置样例自动校验。 |
| Code boundary check | Agent 只能改 ADU 白名单路径。 |
| Test evidence check | PR 缺 build/test/evidence 直接拒绝。 |
| Risk classifier | P0/P1、安全、并发、数据库、硬件关键词自动升级。 |
| Regression matrix | 每个需求自动映射测试覆盖。 |
| L2/L3 classifier | 根据 evidence 判断是否只能 L2。 |

### 8.2 Review Agent 分层

建议在人类前增加 3 类 review Agent：

| Review Agent | 作用 |
| --- | --- |
| Contract Review Agent | 检查字段、API、配置、验收断言是否一致。 |
| Risk Review Agent | 检查并发、内存、安全、数据一致性、回滚风险。 |
| Evidence Review Agent | 检查日志、命令、测试、截图、pcap 是否足以验收。 |

只有 Review Agent 判定为 `human_gate_required=true` 的 PR 才进入人类队列。

### 8.3 人类队列限流

| 队列 | SLA |
| --- | --- |
| P0 发布阻塞 | 2 小时内处理。 |
| P1 高风险 | 6 小时内处理。 |
| Contract 变更 | 每日固定 2 次窗口。 |
| 普通 PR | 由 Review Agent 放行或批量抽检。 |
| 证据补齐 | Evidence Agent 自修，非阻塞不找人。 |

人类不要随时被 Agent 打断，否则人数虽然少了，注意力会被切碎。

## 九、4 人模式下的职责边界

### P1：Agent 工厂负责人 / 总架构

| 负责 | 不负责 |
| --- | --- |
| ADU 队列、优先级、scope、contract freeze、发布决策。 | 普通代码实现、普通编译修复。 |
| 跨 lane 冲突裁决。 | 每个 PR 逐行审查。 |
| L2/L3 最终口径。 | 所有环境排障。 |

### P2：核心网协议与数据面专家

| 负责 | 不负责 |
| --- | --- |
| AMF/SMF/UPF/PFCP/5G-LAN/HA/QoS/互操作高风险审查。 | OAM 页面细节。 |
| 数据面性能、会话、状态同步、回滚风险。 | 普通 CRUD 和文档。 |

### P3：平台、安全与数据专家

| 负责 | 不负责 |
| --- | --- |
| OAM/NMS、达梦、License、加密卡、国密、TLS、ARPF/EIR。 | UPF 数据面细节。 |
| secret、证书、脱敏、供应商 SDK 边界。 | 所有测试执行。 |

### P4：QA / 环境 / 发布证据专家

| 负责 | 不负责 |
| --- | --- |
| CI、UERANSIM/gNBSim/SIPp、Chaos、回归、evidence、验收矩阵。 | 业务架构裁决。 |
| 阻断没有证据的 PR。 | 为每个失败亲自 debug。 |

## 十、最终建议

如果只是问“最少几个人可以维持这套 AI Agent 工厂正常运行”，答案是：

> **3 人。**

如果问“最少几个人可以比较稳地支撑 2.11 全量需求在 6 月 30 日前交付”，答案是：

> **4 人。**

如果问“最少几个人可以支撑尽量全量 L3 真实验收”，答案是：

> **5-6 人。**

建议从 **4 人核心团队** 开始：

1. P1 Agent 工厂负责人 / 总架构 / 发布裁决。
2. P2 核心网协议与数据面专家。
3. P3 平台、安全与数据专家。
4. P4 QA / 环境 / 发布证据专家。

同时设置两个硬规则：

1. Agent PR 没有完整 evidence bundle，不进入人类 review。
2. 外部资源不到位的需求自动 L2，不消耗人类反复联调 HH。

这样可以把原来的 8 人、360-520 HH 压到约 **4 人、240-360 HH**，但前提是 Agent 工厂的自动门禁、Review Agent 和 evidence harness 必须先建好。
