# 核心网产品 2.11 版本需求分析

来源文件：`$HOME/5GC_2.11/核心网产品 2.11 版本功能清单.md`

分析对象：2.11 清单中的 34 条需求。本文按当前 `<workspace-root>` 工作区的 Open5GS/Open5GS-NMS 能力基线进行需求分析，重点识别每条需求的产品目标、现有基础、实现缺口、建议落地范围、验收点和主要风险。

## 一、总体判断

2.11 版本需求可分为六类：

| 类别 | 覆盖需求 | 总体判断 |
| --- | --- | --- |
| 可观测性与运维 | KPI 上报、告警、OAM、LMT、链路检测 | Open5GS 已有 metrics、日志和部分运行态接口基础，但距离面向客户交付的按 TAC/Cell/UPF/网元维度统计、统一告警和北向 OAM 仍需要产品化封装。 |
| 4G/5G 核心能力补齐 | 4G 状态/KPI/告警、4G/5G 互操作 | 5GC 基线较完整，EPC 基础网元存在；互操作涉及 MME/SGW/PGW 与 AMF/SMF/UPF 策略一致性、语音专用承载和跨制式会话连续性，复杂度高。 |
| 5G-LAN 与工业专网 | 配置简化、实现优化、HA、工业增强网关、协议代理 | 当前分支已有 5G-LAN L2、L3 gateway、冗余/HA 相关代码和测试痕迹，是 2.11 中最适合继续深化的方向；关键在配置面、状态面和验收闭环。 |
| 安全与国产化适配 | 国密、加密卡、达梦、ARPF、EIR、信令/流量检测、SM4 隧道 | 多数属于产品化或外部系统适配，需要抽象接口、部署模型、证书/密钥/数据库适配和专门联调环境。 |
| 授权与多核心网运营 | 新架构 License、多核心网分组 | 需要在 NMS/OAM、网元运行态和授权服务之间建立统一的 license enforcement 模型，不能只做 UI 限制。 |
| 架构可靠性 | 多线程、异步同步、备份恢复、内存池优化 | 多为底层架构改造或集群能力，建议独立成专题，避免混入普通功能迭代。 |

建议将 2.11 分为三个落地优先级：

| 优先级 | 建议范围 | 原因 |
| --- | --- | --- |
| 第一优先级 | KPI 上报增强、4G 基础状态/告警/KPI、5G-LAN 配置简化与实现优化、双活 HA、SIM 子网独立配置 | 与既有代码和当前分支能力相关度最高，且直接对应 SP01/SP02 交付目标。 |
| 第二优先级 | License、4G/5G 互操作、QoS 动态生效、OAM、加密卡、达梦 HA/适配、ARPF/EIR | 可落地但依赖跨模块设计、外部系统或客户场景，需要先明确边界。 |
| 第三优先级 | 多线程、zk 同步优化、安全检测、SM4 隧道、工业增强网关、业务数据分析服务器、工业协议代理 | 属于架构级或新产品级需求，建议单独立项和拆分技术预研。 |

## 二、GC_V2.11.0_SP01 需求分析

### 1. KPI 上报增强

**需求描述：**用户数统计（全部/TAC/Cell 维度）、PDU Session 统计（Cell/UPF 维度）、CPU/内存/硬盘实时利用率上报。

**业务目标：**为客户侧运维平台提供可按区域、无线小区、UPF 和系统资源维度观测的实时 KPI，支撑容量评估、故障定位和 SLA 报告。

**当前基础：**

- Open5GS 已有 Prometheus 风格 metrics 和日志体系。
- AMF/SMF/UPF 具备 UE、PDU Session、gNB、PFCP session 等运行态上下文。
- 当前 5G-LAN 分支已有部分 UPF 运行态统计接口思路，可复用为状态导出模式。

**主要缺口：**

- UE 与 PDU Session 统计需要稳定关联 TAC、Cell、UPF、DNN、S-NSSAI 等维度。
- Cell 维度在核心网侧依赖 NGAP/NAS 中保存的 TAI/NR CGI 信息，不能只统计 gNB。
- CPU/内存/硬盘属于主机资源，需要定义由各网元本地采集还是由 OAM/Agent 统一采集。
- 需要统一指标命名、标签基数控制和北向接口格式。

**建议落地范围：**

- AMF 输出 UE 注册数、在线数、按 TAC/Cell/gNB 统计。
- SMF 输出 PDU Session 总量、按 DNN/S-NSSAI/Cell/UPF 统计。
- UPF 输出 PFCP session、N3/N6 流量、丢包、转发错误和 CPU/内存基础状态。
- OAM/NMS 聚合展示并提供北向 REST 或 Prometheus scrape。

**验收点：**

- 两个以上 gNB/Cell 接入时，统计能按 Cell 区分 UE 和 PDU Session。
- UPF 故障或会话迁移后，PDU Session 统计不出现长期重复计数。
- 主机 CPU/内存/硬盘采样周期、单位、时间戳一致。

**风险：**Cell 维度状态若只在 AMF 保存，SMF/UPF 统计需要跨网元关联；建议先由 AMF/SMF 统一维护会话维度映射，UPF 只上报用户面事实指标。

### 2. 4G 核心网基础能力完善

**需求描述：**状态查询（终端/基站/网元）+ 告警 + KPI，与 5G 能力对齐。

**业务目标：**让 EPC 与 5GC 在网管和运维体验上保持一致，避免 4G 场景只能依赖日志排障。

**当前基础：**

- Open5GS 包含 MME、SGW-C、SGW-U、HSS、PCRF 等 EPC 网元。
- 5GC 侧已有较多运行态、metrics 和 NMS 产品化经验可复用。

**主要缺口：**

- 4G 侧 eNB、UE、Bearer、S1/S11/S5/S8、Diameter 等状态需要独立建模。
- 告警需要覆盖 S1AP/SCTP 断链、Diameter 断链、GTP-C/GTP-U 异常、HSS/PCRF 连接异常。
- 4G 与 5G 指标名称、北向字段和 UI 交互需要统一。

**建议落地范围：**

- MME：eNB 状态、UE ECM/EMM 状态、Attach/TAU/Detach 成功率。
- SGW/PGW：Bearer 数、GTP tunnel 数、流量和丢包统计。
- HSS/PCRF：Diameter peer 状态、鉴权/策略请求成功率。
- NMS：4G/5G 使用统一的状态页、告警列表和 KPI 查询模型。

**验收点：**

- eNB 断链、HSS 断链、UE Attach 失败可产生告警并恢复清除。
- 同一 NMS 页面能查询 4G UE 和 5G UE 的在线状态。
- KPI 至少覆盖注册/附着、会话/承载、网元健康、接口链路四类。

**风险：**4G 和 5G 状态机不同，不能强行复用 5G 字段；建议统一展示模型，但底层保持 EPC/5GC 分域适配。

## 三、GC_V2.11.0_SP02 需求分析

### 3. 新架构 License

**需求描述：**用户数限制（不限制基站）、吞吐率限制、多核心网分组管理（CP/UP 分组）。

**业务目标：**支持按商用授权控制核心网容量和多核心网实例，便于交付、扩容和商业计费。

**当前基础：**

- Open5GS 本身没有完整商业 License 框架。
- NMS/OAM 可作为授权配置、展示和告警入口。
- AMF/SMF/UPF 分别掌握用户数、会话数和吞吐率事实数据。

**主要缺口：**

- 授权校验点需要分布在 AMF、SMF、UPF，不能仅靠前端限制。
- 吞吐率限制涉及 UPF 数据面限速和统计窗口。
- 多核心网分组需要定义 CP 组、UP 组、License 与网元实例之间的绑定关系。

**建议落地范围：**

- 建立 license service 或 OAM 授权模块，提供签名授权文件校验。
- AMF 按在线 UE/SUPI 限制接入，SMF 按 PDU Session 或用户数辅助校验，UPF 按总吞吐限制。
- NMS 支持 CP/UP 分组、授权容量、当前使用量和超限告警。

**验收点：**

- 超过用户数限制时，新 UE 注册被拒绝，既有在线 UE 不被误踢。
- 超过吞吐授权时，UPF 能按策略限速或告警。
- 多组 CP/UP 下，授权使用量可按组聚合。

**风险：**License enforcement 涉及客户体验，超限动作需产品确认：拒绝、限速、告警还是宽限期。

### 4. 5G LAN 配置简化

**需求描述：**简化配置步骤；分析当前实现与 3GPP 协议的差异及功能劣势。

**业务目标：**降低 5G-LAN 开通门槛，让 VN Group、DNN、S-NSSAI、UE 成员、N6/TAP/VLAN 等配置形成向导式闭环。

**当前基础：**

- 当前分支已有 5G-LAN Ethernet VN Group、MAC 学习、BUM 转发、IGMP snooping、L3 gateway、N6 TAP 等实现痕迹。
- 既有文档已经将 5G-LAN 控标点整理为 R1/R2/R3/R4 等需求。

**主要缺口：**

- 配置散落在 SMF/UPF YAML、订阅数据、NMS 和测试脚本中。
- 与 3GPP TS 23.501/23.502 的 VN Group、Ethernet PDU Session、NEF/UDM 数据模型差异需要形成明确说明。
- 缺少一键创建 VN Group 后自动校验和状态回读能力。

**建议落地范围：**

- NMS 增加 5G-LAN 配置向导：VN Group、DNN、切片、成员 SUPI、VLAN、网关、限速和 HA 策略。
- 后端生成并应用 SMF/UPF 配置，或通过运行态接口下发。
- 输出协议差异分析表：标准字段、当前实现字段、差异原因、风险和补齐计划。

**验收点：**

- 通过 NMS 创建一个 VN Group 后，两台 UE 能建立 Ethernet PDU Session 并二层互通。
- UI 能展示 VN Group 成员、MAC 表、组播表、BUM 统计和环路告警。
- 文档明确哪些能力是 3GPP 标准、哪些是本地扩展。

**风险：**如果仍依赖重启网元应用配置，配置简化体验会受限；需要明确热加载与重启边界。

### 5. 5G LAN 实现优化

**需求描述：**解决配置复杂、逻辑不连续问题。

**业务目标：**把现有 5G-LAN 从实验/测试能力整理为稳定的产品功能，减少跨模块隐式约定。

**当前基础：**

- UPF 侧已有 VN Group、L2 转发、组播、L3 gateway、HA/冗余等模块。
- SMF 侧已有 HA orchestration 和 PFCP session 建立路径。

**主要缺口：**

- Ethernet PDU 标识、VN Group membership、PFCP PDR/FAR/QER 和 N6 行为之间容易出现跨进程状态不一致。
- 配置、控制面、数据面、状态面缺少统一生命周期。
- 自动化回归需要覆盖单播、广播、组播、VLAN、L3 gateway、HA 切换等组合场景。

**建议落地范围：**

- 梳理 `VN Group -> UE member -> PDU Session -> UPF forwarding state` 的唯一数据流。
- 将 Ethernet PDU、VN Group ID、VLAN 策略、BUM 策略等通过标准或显式扩展字段传递。
- 增加一致性检查：配置成员与运行态 session、MAC 表、PFCP 规则一致。

**验收点：**

- 5G-LAN 端到端回归全部自动化，失败时能定位到配置、控制面或数据面。
- 新增 UE、删除 UE、修改 VLAN 或网关配置后，运行态一致性可检查。
- 非 IP Ethernet 帧不会被误判为异常 IP 包。

**风险：**当前分支功能较多，建议先固化最小可交付路径，再处理高级冗余和工业扩展。

### 6. 双活模式 HA

**需求描述：**支持基于双活模式的主备倒换。

**业务目标：**在工业专网中提升用户面可靠性，主 UPF 故障时业务快速恢复，或通过双活转发降低单点风险。

**当前基础：**

- 当前分支已有 UPF HA group、active/standby、PFCP session 同步、shadow state、冗余窗口等代码痕迹。
- 既有 R3/R4 文档已讨论主备切换和双发选收。

**主要缺口：**

- 双活、主备、双发选收是三种不同模型，需要先定义产品模式。
- 会话状态、TEID、PDR/FAR/QER、MAC 表、组播表、L3 gateway NAT/ARP 状态都可能需要同步。
- N3/N6 地址漂移或双路径路由需要外部网络配合。

**建议落地范围：**

- SP02 先交付主备快速倒换：主 UPF 故障检测、备 UPF 接管、状态同步、业务中断时间统计。
- 双活作为增强：同一业务双路径转发时，明确去重位置和序列号来源。
- NMS 提供 HA 组、角色、同步延迟、切换事件和健康状态。

**验收点：**

- 主 UPF 进程或链路故障后，已建立 PDU Session 能恢复转发。
- 记录中断时长、丢包数、状态同步延迟。
- HA 切换后 MAC 表和 L3 gateway 状态不丢失或可快速重建。

**风险：**如果没有 RAN、路由器、VRRP/BGP 或双 N3/N6 网络配合，单靠核心网软件无法保证真实双活。

### 7. 4G/5G 互操作

**需求描述：**4 切 5、5 切 4，语音切换（专用承载）+ 数据切换（默认承载）业务不中断。

**业务目标：**支持 EPS 与 5GS 间移动性，保证数据会话和语音业务连续。

**当前基础：**

- Open5GS 同时具备 EPC 与 5GC 网元。
- 代码中具备 MME/SGW/PGW 与 AMF/SMF/UPF 的基础协议栈。

**主要缺口：**

- 需要 N26 或无 N26 互操作方案选择。
- 语音切换依赖 IMS、QCI/5QI 映射、专用承载、P-CSCF、EPS fallback 或 VoNR/VoLTE 策略。
- 数据不中断涉及 IP 地址保持、PGW-C/SMF/UPF 锚点策略和会话上下文迁移。

**建议落地范围：**

- 先定义支持场景：4G 到 5G 数据会话连续、5G 到 4G EPS fallback、语音专用承载保留。
- 建立 QCI 与 5QI、APN 与 DNN、TAI 与 TAI/TAI list、UE subscription 的映射。
- 与 IMS/SBC 明确边界：核心网负责承载和策略，IMS 负责 SIP/媒体控制。

**验收点：**

- UE 从 5G 覆盖移动到 4G 后，默认数据业务不中断或中断时间满足目标。
- 语音业务触发专用承载并在切换后保持。
- 失败路径有明确 cause 和告警。

**风险：**需要真实或仿真的双制式 RAN/UE/IMS 环境，单元测试无法充分证明业务不中断。

### 8. QoS 动态生效

**需求描述：**工业协议识别模块触发 MQTT -> UMS -> 核心网，实时修改 QoS，无需 UE 下次注册。

**业务目标：**工业业务流识别后即时提升或调整 QoS，满足低时延、高可靠或特定带宽策略。

**当前基础：**

- Open5GS 有 PCF/SMF/UPF 的策略和 PFCP QER/PDR/FAR 机制。
- SBI OpenAPI 模型中存在 QoS、policy、AF event 等基础概念。

**主要缺口：**

- 外部工业协议识别模块、MQTT、UMS 与核心网接口协议未定义。
- 动态 QoS 需要触发 SM policy update、SMF 修改 PFCP QER/PDR，并通知 AMF/RAN 更新 QoS Flow。
- 需要处理策略冲突、回滚、超时恢复和审计。

**建议落地范围：**

- 定义北向 `QoSPolicyUpdate` API：SUPI/GPSI、DNN、S-NSSAI、应用流五元组、目标 5QI/GFBR/MFBR、有效期。
- SMF 接收策略后执行 Session Modification，更新 UPF QER/PDR，必要时触发 N2 SM QoS 修改。
- NMS 展示动态策略来源、命中会话、生效时间和恢复动作。

**验收点：**

- UE 在线时下发 QoS 变更，不重新注册即可看到 QER 或 QoS Flow 更新。
- 策略过期后自动恢复默认 QoS。
- 无匹配会话、冲突策略、UPF 修改失败均有明确返回和告警。

**风险：**只修改 UPF QER 可能无法满足无线侧 QoS；若需要 RAN 侧资源保障，必须走完整 N2 SM 修改。

### 9. SIM 子网独立配置

**需求描述：**5G SIM 子网在网管界面直接添加，不依赖全局地址池。

**业务目标：**按 SIM/用户/客户独立配置 IP 地址池，适配工业客户隔离和快速开卡。

**当前基础：**

- Open5GS 支持按 DNN/S-NSSAI 配置 session subnet。
- 当前分支已有 DNN/APN 自适应能力，可作为地址池策略扩展参考。
- NMS/DB 中已有 subscriber 配置入口。

**主要缺口：**

- 需要在订阅数据或策略数据中表达 per-SIM/per-group 地址池。
- SMF 地址分配逻辑需支持优先级：SIM 独立池 > DNN/S-NSSAI 池 > 全局默认池。
- NMS 需要校验地址池重叠、容量和绑定关系。

**建议落地范围：**

- NMS subscriber 页面增加 SIM 独立子网字段和批量导入能力。
- SMF 增加按 SUPI/GPSI/用户组选择地址池逻辑。
- 地址池状态支持已用/可用/冲突查询。

**验收点：**

- 同一 DNN 下，不同 SIM 可分配到不同独立子网。
- 子网耗尽时会话建立失败原因明确，且不误用全局地址池，除非策略允许 fallback。
- 修改 SIM 子网后，新会话生效，旧会话行为符合产品定义。

**风险：**地址池变更对在线会话是否生效必须明确，建议默认只对新 PDU Session 生效。

### 10. 江南信安加密卡适配

**需求描述：**适配 PCIE 接口硬件加密卡。

**业务目标：**满足安全专网对国产密码模块和硬件密钥保护的要求。

**当前基础：**

- Open5GS 有 SUCI、鉴权、TLS、密钥配置等敏感数据处理路径。
- 旧控标点需求中已识别 HSM/密码机抽象层需求。

**主要缺口：**

- 缺少硬件加密卡 SDK 抽象层和错误处理。
- 需要明确加密卡用于 SUCI 解密、AKA 向量生成、数据库字段加密、TLS 私钥操作，还是全部覆盖。
- 运行环境、驱动、权限、容器化部署方案需要定义。

**建议落地范围：**

- 先做 crypto provider 抽象：software、PKCS#11、江南信安 SDK 三类后端。
- 第一阶段接入 SUCI 私钥/敏感字段加解密；第二阶段评估鉴权算法硬件化。
- NMS 展示设备状态、密钥槽位、健康检查和错误告警。

**验收点：**

- 加密卡在线时可完成指定密码操作。
- 加密卡故障时系统按策略降级、拒绝或告警，不崩溃。
- 密钥不以明文落盘，日志不打印敏感材料。

**风险：**厂商 SDK 许可、驱动和硬件环境是主要阻塞项；建议先用接口模拟器开发。

### 11. 达梦数据库 HA 适配

**需求描述：**达梦数据库支持 HA 部署，提升安全专网可靠性。

**业务目标：**替换或补充 MongoDB，满足国产数据库和高可用部署要求。

**当前基础：**

- Open5GS 当前主要依赖 MongoDB/libmongoc 进行订阅数据存储。
- NMS 也通常依赖 MongoDB 模型。

**主要缺口：**

- 数据访问层与 MongoDB 耦合较强，直接替换数据库成本高。
- 达梦 HA 的连接串、故障切换、事务语义、索引和 schema 迁移需要适配。
- 需要明确是核心网运行库、NMS 库，还是审计/配置库迁移到达梦。

**建议落地范围：**

- 先做 NMS/OAM 配置库达梦适配，再评估核心网订阅库抽象。
- 建立 repository/DAO 层，避免业务代码直接依赖 MongoDB 文档结构。
- 提供达梦主备连接配置、健康检查、故障切换重连和迁移脚本。

**验收点：**

- 达梦主库故障后，NMS/OAM 能恢复读写。
- subscriber、VN Group、license、告警等关键数据读写一致。
- 数据库切换过程不会导致核心网进程崩溃。

**风险：**若要求核心网实时订阅库完全达梦化，影响范围很大，应单独立项。

### 12. OAM 模块开发

**需求描述：**独立 OAM 管理模块，适配 LMT，北向接口从 OAM 出。

**业务目标：**形成统一运维入口，承接配置、状态、告警、KPI、北向接口和 LMT 对接。

**当前基础：**

- Open5GS-NMS/WebUI 可提供部分配置管理。
- 各网元已有日志、metrics 和配置文件。

**主要缺口：**

- OAM 与 NMS、LMT、网元本地配置之间的职责边界未定义。
- 北向接口需要统一鉴权、审计、限流、版本管理。
- 配置下发需要支持校验、应用、回滚和状态确认。

**建议落地范围：**

- OAM 作为统一后端：配置管理、KPI 聚合、告警管理、北向 API。
- LMT 作为本地维护终端，通过 OAM API 操作，不直接改网元配置。
- 网元侧提供标准状态/metrics/config apply 接口。

**验收点：**

- OAM 能查询所有网元健康状态、版本、配置摘要。
- 北向 API 可查询 KPI、告警、UE/session 状态。
- 配置变更有审计、回滚和应用结果。

**风险：**OAM 是平台级能力，不建议和单个 5G-LAN 功能混在一个迭代中实现。

### 13. ARPF 独立部署

**需求描述：**主认证增强系统独立部署，满足密码使用和加密算法安全要求。

**业务目标：**将认证密钥和算法从 UDM/AUSF 中解耦，满足高安全部署和国产密码要求。

**当前基础：**

- Open5GS 有 AUSF/UDM/UDR 鉴权流程。
- SBI TLS 和 NF discovery 基础存在。

**主要缺口：**

- 3GPP 中 ARPF 通常属于 UDM 内部功能，独立部署需要定义服务边界和接口。
- AUSF/UDM 与外部 ARPF 的鉴权向量请求、错误码、超时和重试需设计。
- 密钥材料迁移、加密存储和审计要求高。

**建议落地范围：**

- 定义 UDM 到独立 ARPF 的内部 SBI/REST/gRPC 接口。
- ARPF 提供鉴权向量生成、密钥管理、算法插件和审计日志。
- 支持软件算法与硬件加密卡 provider。

**验收点：**

- UE 注册鉴权路径通过独立 ARPF 获取向量。
- ARPF 故障时注册失败原因明确，系统产生告警。
- 密钥访问有审计，敏感字段不出现在日志。

**风险：**独立 ARPF 涉及协议非标准化或私有扩展，需要与客户安全规范确认。

### 14. EIR 独立部署

**需求描述：**IMEI 白名单 + 机卡绑定功能独立部署，通过 SBI 接口与 5GC 通信。

**业务目标：**独立管理设备准入，支持 IMEI 白名单、黑名单和 SUPI-IMEI 绑定。

**当前基础：**

- Open5GS AMF 可获取 PEI/IMEISV，并可在注册流程中做策略判断扩展。
- SBI OpenAPI 模型中包含 5G_EIR NF type。

**主要缺口：**

- 缺少 EIR 网元、数据库模型、SBI 查询接口和 AMF 调用路径。
- 需要定义白名单、黑名单、灰名单和机卡绑定冲突策略。
- 在线设备被禁用后是否主动去注册需要明确。

**建议落地范围：**

- 新增独立 EIR 服务或 OAM 子模块，提供 PEI/SUPI 查询。
- AMF 注册流程调用 EIR，按结果允许、拒绝或告警。
- NMS 提供 IMEI 管理、机卡绑定、导入导出、在线踢出。

**验收点：**

- 非白名单 IMEI 注册被拒绝。
- SUPI 与错误 IMEI 组合注册被拒绝并告警。
- 删除在线设备后，AMF 触发 deregistration 或按策略保留到下次注册。

**风险：**AMF 注册时同步调用外部 EIR 会影响注册时延；需缓存和超时策略。

## 四、IMS_V2.11.0_SP01 需求分析

### 15. 银河麒麟 V10 操作系统适配

**需求描述：**IMS 在银河麒麟 V10 上适配，状态为转测。

**业务目标：**满足国产操作系统部署要求。

**Open5GS 相关性：**该需求主体是 IMS，但核心网侧也可能需要统一安装、服务管理、日志路径和 OAM 对接风格。

**建议落地范围：**

- 明确核心网与 IMS 的 OS 适配基线：内核版本、systemd、依赖包、编译器、OpenSSL/国密库。
- 对 Open5GS 侧输出麒麟 V10 部署检查清单，保证与 IMS 同环境部署时不会冲突。

**验收点：**

- IMS 服务在麒麟 V10 上完成安装、启动、重启、日志轮转和基础呼叫验证。
- 与核心网对接的接口地址、证书和 DNS 配置稳定。

**风险：**如果核心网也要求同 OS 交付，需要额外验证 Open5GS 编译和运行依赖。

### 16. 国密 UDM/国密 USIM 卡对接

**需求描述：**国密 UDM 与国密 USIM 卡对接。

**业务目标：**支持国产密码算法的用户鉴权和隐私保护。

**当前基础：**

- Open5GS 有 UDM/AUSF/SUCI/ZUC 相关基础。
- 旧控标点中已识别 SM2/SM3/SM4、国密 AKA 和国密 USIM 的缺口。

**主要缺口：**

- 国密 USIM 鉴权算法、卡参数、向量生成流程和标准/私有协议边界需要确认。
- UDM/AUSF 与 ARPF/加密卡的职责需统一。

**建议落地范围：**

- 先输出算法与接口适配方案：国密 SUCI 解密、鉴权向量生成、卡数据管理。
- 建立算法 provider 插件，支持软算法和硬件加密卡。
- 增加国密 USIM 互通测试用例。

**验收点：**

- 国密 USIM 可完成注册鉴权。
- 普通 USIM 与国密 USIM 可按用户配置并存。
- 鉴权失败 cause 明确且可审计。

**风险：**国密 AKA 可能涉及私有实现，需要客户、卡商和加密卡厂商共同确认。

### 17. 达梦数据库适配

**需求描述：**IMS 达梦数据库适配。

**Open5GS 相关性：**与 GC 侧达梦 HA 适配类似，但主体是 IMS 数据库。若 OAM 统一管理，需要统一数据库连接、备份、审计和告警模型。

**建议落地范围：**

- IMS 独立完成达梦 schema、连接池、主备切换。
- OAM 只接入数据库健康状态和告警，不直接耦合 IMS 业务表。

**验收点：**

- 达梦主备切换后 IMS 基础业务恢复。
- OAM 能展示 IMS 数据库状态。

**风险：**不要将 IMS 达梦适配与 5GC MongoDB 替换混为同一任务。

### 18. IMS 基础告警功能

**需求描述：**IMS 基础告警。

**业务目标：**补齐 IMS 网元、接口、呼叫业务和资源状态告警。

**Open5GS 相关性：**需要与核心网告警模型统一，便于 NMS/OAM 一屏展示。

**建议落地范围：**

- 定义统一告警字段：级别、网元、接口、对象、首次发生、恢复时间、建议动作。
- IMS 输出 SIP trunk、P-CSCF/S-CSCF/I-CSCF、数据库、媒体资源等告警。
- OAM 聚合展示，与 4G/5G 告警使用同一北向格式。

**验收点：**

- IMS 网元断链、数据库异常、呼叫失败率过高可产生和恢复告警。
- 告警可北向上报并可确认/清除。

**风险：**告警来源如果只靠日志解析，稳定性较差；建议网元主动上报。

### 19. LMT 优化（跟随核心网风格）

**需求描述：**IMS LMT 界面和交互跟随核心网风格。

**业务目标：**统一本地维护体验，降低现场运维学习成本。

**Open5GS 相关性：**应以核心网 OAM/LMT 的导航、权限、状态、告警和配置交互作为统一规范。

**建议落地范围：**

- 输出 LMT 交互规范：登录、网元状态、配置编辑、告警、日志下载、诊断工具。
- IMS 和核心网复用组件风格和 API 鉴权。

**验收点：**

- 相同类型功能在 IMS LMT 与核心网 LMT 中字段、按钮和操作结果一致。
- LMT 操作有审计记录。

**风险：**UI 风格统一不能替代底层接口统一，建议先统一 API 和数据模型。

## 五、未分配版本需求分析

### 20. 5GC 单线程 -> 多线程

**需求描述：**提升整体性能，突破用户数处理瓶颈。

**业务目标：**提升 AMF/SMF/UPF 控制面或数据面并发能力。

**当前基础：**Open5GS 是事件驱动架构，部分组件使用 poll/timer/FSM 和内存池。

**主要缺口：**

- 多线程会影响全局 context、hash/list、timer、FSM、SBI/PFCP/NGAP 处理顺序。
- 需要先定位瓶颈网元和瓶颈路径，不能泛泛改全局多线程。

**建议落地范围：**

- 先做性能 profiling：注册、PDU Session 建立、PFCP、UPF 转发分别测量。
- 选择单一网元试点，例如 AMF SBI/NGAP 分发或 SMF session worker。
- 明确线程模型：acceptor + worker、per-UE affinity、per-session affinity。

**验收点：**

- 指定用户规模下注册/会话建立 TPS 提升。
- 并发测试无 UE/session 状态错乱。
- 退出、重启、异常路径无内存破坏。

**风险：**这是架构级改造，风险高，必须独立设计和长期压测。

### 21. zk 同步写 -> 异步写

**需求描述：**提升主备同步速度。

**业务目标：**降低主备同步对业务路径的阻塞。

**当前基础：**当前 Open5GS 基线不以 ZooKeeper 作为核心依赖；该需求可能来自产品自研 HA 框架。

**主要缺口：**

- 需确认 zk 在当前产品中的用途：配置、会话、主备状态、告警还是 license。
- 异步写需要处理丢失、乱序、幂等和一致性窗口。

**建议落地范围：**

- 先梳理同步写路径和延迟占比。
- 将非关键状态改为异步队列，关键会话状态保留确认或使用 WAL。
- 增加同步积压、失败重试和一致性检查指标。

**验收点：**

- 主业务路径延迟下降。
- 主备切换后状态丢失量在产品目标内。
- zk 故障/恢复后队列可恢复或明确降级。

**风险：**异步化可能换来数据一致性风险，必须按数据类型分级。

### 22. 单机备份与恢复

**需求描述：**故障后快速恢复业务。

**业务目标：**支持单节点部署下的配置、订阅、license、告警和关键运行数据备份恢复。

**当前基础：**

- Open5GS 配置主要为 YAML，订阅数据在 MongoDB，NMS 有自身数据。

**主要缺口：**

- 需要定义备份范围、频率、加密、压缩、校验和恢复流程。
- 在线运行态会话通常不适合完整备份恢复，需明确只恢复配置/订阅还是也恢复会话。

**建议落地范围：**

- OAM 提供备份包：配置文件、数据库 dump、license、证书、NMS 配置。
- 支持手动备份、定时备份、恢复前校验和恢复后健康检查。

**验收点：**

- 新机器恢复备份后，核心网能启动并完成 UE 注册/PDU Session。
- 备份包损坏、版本不匹配、证书缺失能提前报错。

**风险：**敏感数据备份必须加密，并与加密卡/HSM 策略兼容。

### 23. zk 内存池优化

**需求描述：**减少小内存碎片；新增 OMU 主备上下文比较/清除命令。

**业务目标：**提升长期运行稳定性，并提供主备状态维护工具。

**当前基础：**Open5GS 自身有内存池机制，但 zk/OMU 更像产品外层集群模块。

**主要缺口：**

- 需确认内存碎片发生在 zk 客户端、OMU 进程还是共享状态模块。
- 主备上下文比较需要定义对象范围和一致性规则。

**建议落地范围：**

- 对小对象分配热点做内存 profile。
- 引入固定大小 slab/pool 或对象复用。
- OMU 增加 context diff、context clear、dry-run 和审计。

**验收点：**

- 长稳测试中内存碎片和 RSS 增长可控。
- 主备 context diff 能指出缺失、冲突和版本差异。
- clear 命令有权限控制和二次确认。

**风险：**清除主备上下文是高危运维动作，必须防误操作。

### 24. SBI 支持 TLS2.0

**需求描述：**UDM 与 ARPF 分离时 SBI 接口加密。

**业务目标：**保护 UDM 与独立 ARPF 间的鉴权和密钥相关交互。

**说明：**业界常见表述为 TLS 1.2/1.3；清单中的 TLS2.0 需要产品侧确认是否指 TLS 1.2 及以上、双向 TLS，或内部版本名称。

**当前基础：**

- Open5GS SBI 基于 HTTP/2 REST，配置中已有 TLS 证书材料。

**主要缺口：**

- UDM-ARPF 分离接口尚未定义。
- 需要 mTLS、证书轮换、国密 TLS 或普通 TLS 的选择。

**建议落地范围：**

- 明确目标为 SBI mTLS 1.2+ 或国密 TLS。
- UDM 与 ARPF 互相校验证书，支持证书过期告警和热更新。
- OAM 统一管理证书和信任链。

**验收点：**

- 无证书、错误证书、过期证书均无法访问 ARPF。
- 正确证书下 UE 鉴权成功。
- 证书轮换不导致长期业务中断。

**风险：**如果客户要求国密 TLS，需额外密码库和互通测试。

### 25. 5GC 内置信令分析模块

**需求描述：**内置信令分析。

**业务目标：**发现异常注册、异常会话、频繁切换、失败率升高等安全或质量问题。

**当前基础：**

- AMF/SMF/MME 等网元有事件和日志。
- OpenAPI 模型中已有异常行为、事件订阅等概念可参考。

**主要缺口：**

- 需要从日志解析转为结构化事件上报。
- 需要规则引擎、阈值、窗口、告警和证据保存。

**建议落地范围：**

- 网元输出结构化信令事件：注册、鉴权、PDU Session、切换、释放、失败 cause。
- 分析模块按 SUPI/gNB/Cell/DNN/IP 维度聚合。
- 支持规则：失败率、频率、异常 cause、异常位置变化。

**验收点：**

- 模拟大量注册失败时产生异常告警。
- 可追溯到具体 UE、gNB、时间窗口和 cause 分布。

**风险：**全量信令事件可能带来高存储和隐私风险，需要采样和脱敏。

### 26. 5GC 内置违规外联流量检测与分析

**需求描述：**检测 UE 访问违规外部地址/端口/域名并分析。

**业务目标：**满足安全专网对外联审计和阻断要求。

**当前基础：**

- UPF 掌握 N3/N6 用户面流量。
- 旧控标点中已有违规外联检测、告警与阻断需求。

**主要缺口：**

- UPF 当前不是完整 DPI/IDS，深度解析能力有限。
- 域名检测需要 DNS/TLS SNI/HTTP Host 解析，越来越受加密影响。
- 阻断规则需要与 PDR/FAR/QER 或外部防火墙联动。

**建议落地范围：**

- 第一阶段做五元组/IP 段/端口黑白名单检测与阻断。
- 第二阶段接入镜像到外部安全分析模块，UPF 只做流量导出和阻断执行。
- NMS 提供策略、告警和证据留存。

**验收点：**

- 命中黑名单目的 IP/端口时产生告警并阻断。
- 白名单策略优先生效。
- 告警包含 SUPI、DNN、源/目的、时间、规则 ID。

**风险：**在 UPF 内做复杂 DPI 会影响转发性能，建议边界清晰。

### 27. 5GC 内置 N3 接口 SM4 隧道加密

**需求描述：**N3 接口 SM4 隧道加密，需基站配合。

**业务目标：**保护 gNB 与 UPF 间 N3 用户面流量。

**当前基础：**

- 标准 N3 使用 GTP-U；Open5GS UPF 处理 GTP-U。

**主要缺口：**

- SM4 隧道是非标准或专有扩展，需要 gNB/基站侧配合。
- 加密位置、密钥协商、重放保护、MTU、性能损耗都需设计。

**建议落地范围：**

- 明确是 GTP-U payload 加密、GTP-U 外层隧道加密，还是 IPsec/国密 VPN。
- 优先考虑外置国密 VPN 网关，核心网只管理策略和状态。
- 若内置 UPF，需要 crypto provider 和硬件加速。

**验收点：**

- 开启加密后，gNB-UPF 用户面可正常收发。
- 抓包不可见明文业务数据。
- 密钥错误、隧道断开有告警。

**风险：**内置加密会显著影响 UPF 性能，且破坏标准互通性。

### 28. 链路检测 ping 功能

**需求描述：**核心网 -> 基站/业务服务器方向链路检测。

**业务目标：**现场快速判断 N2/N3/N6 及业务服务器连通性。

**当前基础：**

- OAM/NMS 可作为操作入口。
- 网元所在主机可执行 ICMP/TCP/UDP 探测。

**主要缺口：**

- 需要明确从哪个网元/接口发起探测：AMF 到 gNB、UPF 到业务服务器、OAM 到网元。
- 需要权限控制，避免 ping 被滥用。

**建议落地范围：**

- OAM 提供链路检测任务：目标、源网元、源接口、协议、次数、超时。
- 网元或 agent 执行探测并返回时延、丢包、错误原因。
- 支持结果历史和告警联动。

**验收点：**

- 可从 AMF 主机检测 gNB SCTP 地址连通性。
- 可从 UPF N6 接口检测业务服务器。
- 目标不可达时返回明确原因。

**风险：**普通 ICMP ping 不能证明 SCTP/GTP-U/业务协议可用，建议支持 TCP/UDP 探测扩展。

### 29. 独立工业增强网关

**需求描述：**双发选收/负载分担/切换/VxLAN L3 转 L2。

**业务目标：**为工业场景提供核心网外的增强可靠性与协议适配能力。

**当前基础：**

- 当前分支已有 UPF 冗余、L3 gateway、5G-LAN 等相关能力，但工业增强网关更像独立产品。

**主要缺口：**

- 双发选收需要端到端序列号和去重协议。
- VxLAN L3 转 L2 涉及外部网络、桥接、路由和安全隔离。
- 负载分担/切换策略需要与 UE、CPE、RAN、UPF 协同。

**建议落地范围：**

- 独立立项为边缘网关产品，Open5GS 提供会话、策略、状态和链路质量接口。
- 网关负责 VxLAN、双发选收、负载分担和工业协议适配。

**验收点：**

- 双链路下重复包可去重，单链路故障业务不中断或快速恢复。
- VxLAN L3/L2 转换符合目标拓扑。

**风险：**不建议全部塞入 UPF，否则会扩大核心网数据面复杂度。

### 30. 用户业务数据分析服务器

**需求描述：**7 层解析 + AI。

**业务目标：**对用户业务流进行应用层识别、质量分析和智能告警。

**当前基础：**

- UPF 可作为流量镜像或元数据导出点。

**主要缺口：**

- 7 层解析与 AI 分析超出核心网主流程。
- 数据隐私、加密流量、存储规模和模型准确率需要专项设计。

**建议落地范围：**

- 独立服务器接收 UPF mirror/NetFlow/IPFIX/五元组日志。
- 核心网提供流量导出策略、用户映射和阻断回调接口。

**验收点：**

- 能识别指定应用协议或工业协议样本。
- 分析结果能关联 SUPI/DNN/Cell/UPF。
- 可向 OAM 产生告警或策略建议。

**风险：**AI 结果不能直接自动阻断，建议先人工确认闭环。

### 31. 工业协议代理

**需求描述：**降低空口传输抖动造成的工业协议超时。

**业务目标：**通过代理、缓存、重传或协议适配减轻无线链路抖动对工业控制协议的影响。

**当前基础：**

- 核心网可提供 QoS、会话和链路状态，但不直接理解工业协议。

**主要缺口：**

- 需要明确目标协议，例如 Modbus、Profinet、OPC UA、EtherNet/IP 等。
- 代理位置可能在 CPE、边缘网关或业务侧，不宜放在 AMF/SMF。

**建议落地范围：**

- 作为边缘应用或工业增强网关能力实现。
- 核心网提供 QoS 动态调整、链路状态和策略接口。

**验收点：**

- 指定工业协议在抖动/丢包环境下超时率下降。
- 代理行为不破坏协议一致性和安全策略。

**风险：**不同工业协议时序和一致性要求差异大，需要按协议逐个验证。

## 七、建议交付拆分

### SP01 最小交付包

- KPI 上报增强：完成 UE、PDU Session、UPF、主机资源基础指标。
- 4G 核心网基础能力：完成 EPC 网元状态、基础 KPI 和关键链路告警。
- 输出 KPI/告警北向接口规范，作为后续 OAM 输入。

### SP02 核心交付包

- 5G-LAN 配置简化与实现优化：NMS 配置向导、运行态回读、协议差异分析。
- 双活/主备 HA：先交付可验证的主备切换和状态展示。
- SIM 子网独立配置：NMS 配置 + SMF 地址池选择。
- License：完成用户数限制和使用量统计，吞吐限制可作为增强项。

### 专项预研包

- 4G/5G 互操作：单独建立 RAN/UE/IMS 测试环境。
- 国密/加密卡/ARPF/EIR：建立安全架构设计和厂商联调计划。
- 达梦 HA/适配：先从 OAM/NMS 数据库适配开始。
- 多线程、zk 异步、备份恢复、安全检测、工业增强网关：单独立项评审。
