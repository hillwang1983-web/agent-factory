# 核心网产品 2.11 全量需求代码量评估

来源：

- `docs/core-network-2.11-0630-full-scope-agent-plan.md`
- `docs/core-network-2.11-ai-agent-effort-estimate.md`
- 当前 `<workspace-root>` 工作区 Open5GS/Open5GS-NMS/UERANSIM 结构

评估口径：

- 目标是 **2026-06-30 前完成全部 34 条需求**。
- 代码量按 AI Agent 工厂拆分后的交付形态估算，不按人月、人日估算。
- 数字为净新增/修改有效代码量区间，单位为 LOC。
- LOC 包含核心网 C/C++、NMS/OAM 后端、前端、测试/仿真、脚本/部署、配置样例。
- 不把生成后的 `dist/`、`node_modules/`、编译产物、长日志、截图和纯文档计入代码量。
- 外部依赖不到位的需求按 L2 mock/provider/simulator 路径估算；若要 L3 真实硬件/现场验收，代码量可能增加 20%-60%，主要增加在适配、诊断和联调脚本。

## 一、总体代码量

### 1.1 总量区间

| 交付等级 | 代码量区间 | 说明 |
| --- | ---: | --- |
| L2 软件侧全量完成 | 78k-123k LOC | 全部需求有代码闭环；硬件/现场依赖用 mock、provider、simulator、容器环境完成。 |
| L3 尽量真实验收 | 103k-168k LOC | 外部硬件、达梦、国密 USIM、基站 SM4、真实 RAN/UE/IMS 尽量接入，增加适配和诊断代码。 |
| 风险上限 | 180k+ LOC | 若 OAM 做成完整平台、DPI 做深度解析、多线程扩大到多个网元、工业网关做商用品质，会超过此量级。 |

推荐按 **L2 软件侧全量完成 + 关键项 L3 补证** 规划，即 **约 95k-140k LOC**。

## 四、总量校正

逐需求简单相加会得到较高数值，因为 OAM 基座、告警模型、测试框架、provider、前端组件、部署脚本会在多个需求间复用。按共享组件去重后，推荐用以下总控：

| 口径 | 代码量 |
| --- | ---: |
| 逐需求未去重合计 | 155k-292k LOC |
| 共享组件去重后 L2 | 78k-123k LOC |
| 共享组件去重后 L3 | 103k-168k LOC |
| 建议计划控制线 | 95k-140k LOC |

## 六、代码量风险项

| 风险 | 代码量影响 | 控制方式 |
| --- | --- | --- |
| OAM 做成完整平台 | +15k-35k LOC | 只做状态、指标、告警、北向、最小 UI。 |
| DPI 做深度应用层解析 | +20k-60k LOC | 6/30 前只做五元组/端口/IP 段和插件接口。 |
| 多线程扩大到多个网元 | +15k-40k LOC | 只做 AMF 或 SMF 单路径 feature flag 试点。 |
| 工业增强网关做商用品质 | +20k-50k LOC | 只做独立服务 skeleton + VxLAN/双发选收 PoC。 |
| 达梦完全替换核心 MongoDB | +20k-45k LOC | 先做 DAO/adapter 与关键表迁移，不全量替换。 |
| 国密 TLS/硬件卡真实 SDK 差异大 | +5k-20k LOC | provider interface + SDK adapter 分层，mock 与真实 adapter 分离。 |

## 七、最终建议

按 6 月 30 日全量交付计划拆分后，代码量不是“几十个小改动”，而是一个 **约 95k-140k LOC 的高并发软件侧交付包**。其中核心网主干 C/C++ 约占 30%-40%，测试/仿真/证据代码约占 25%-30%，OAM/NMS 约占 30%-35%。

如果要求尽量 L3 真实验收，建议把总控上限放到 **160k LOC 左右**，并把超出部分严格限制在硬件/数据库/RAN/IMS 适配层，不能让 OAM 平台化、DPI 深解析和多线程大改吞掉交付窗口。
